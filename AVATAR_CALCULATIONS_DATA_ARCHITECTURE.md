# Логическая архитектура сбора данных и расчетов (AVATAR v2.1)

В данном документе описана полная логическая архитектура сбора данных пользователя, выполнения астрологических расчетов и формирования итогового датасета (Digital Soul Blueprint).

## 1. Сбор данных (Frontend)
### 1.1. Onboarding Flow
1. Пользователь вводит параметры рождения:
   - Дата рождения (DD.MM.YYYY)
   - Время рождения (HH:MM)
   - Место рождения (Город)
2. Приложение использует **Google Places API** (автокомплит) для получения расширенных геоданных:
   - Название города (`city_name`)
   - Широта (`lat`) / Долгота (`lon`)
   - Часовой пояс (`timezone`)
3. Отправка POST-запроса на `/api/auth/profile` со всеми собранными данными (дата и время рождения, широта, долгота, зона и локация).

## 2. Инициализация расчета (Backend)
### 2.1. Валидация геоданных (Geocoding Cache)
Фоновая задача на сервере принимает запрос и сверяет данные местоположения.
- Выполняется запрос к слою кэширования: `SELECT lat, lon, timezone FROM geocode_cache WHERE city_name = ?`
- **Hit (кэш):** Используются полученные координаты и таймзона из таблицы.
- **Miss (мимо кэша):** Кэш обновляется по результатам геокодинга через Nominatim или приходят напрямую с клиента; данные сохраняются в БД.
- Преобразование локального времени (с помощью `pytz.timezone`) в UTC для точных астрономических вычислений (`Julian Day`).

## 3. Слой расчетов (Layer 1 - Calculators)
### 3.1. Ядро вычислений (pyswisseph / `natal_chart.py`)
Интеграция с библиотекой швейцарских эфемерид `pyswisseph`. Запускается распараллеленная (batch) выборка данных:
1. **Julian Day**: Преобразование даты/времени в юлианский формат для расчетов.
2. **Система Домов**: Основная система плацидус (`Placidus`). При ошибках для экстремальных широт — фоллбэк на систему целого знака (`Whole Sign`).
3. **Планеты**: Расчет долготы, позиции в знаке, градуса, скорости и фазы ретроградности/стационарности.
4. **Аспекты и элементы**: Вычисление связей, квадрантов, стихий, модальностей.

### 3.2. Выходной формат (Rich JSON)
Возвращается строго типизированный `Rich JSON`, содержащий:
- Метаданные (система домов, тип узлов).
- Планеты с учетом их фазы и координат (`longitude, sign, house, retrograde, speed`).
- Вершины/Куспиды 12 домов и углы (Ascendant, Midheaven).

## 4. Слой интерпретации (Layer 2 - Interpreters)
### 4.1. Формирование динамических запросов (Query Builder)
Агент `western_astrology_agent.py` анализирует сгенерированный `Rich JSON` и строит матрицу запросов:
- **Безусловные цели**: Позиции всех личных планет.
- **Условные цели**: Социальные и высшие планеты в их значимых позициях (в угловых домах или с высоким баллом достоинства).
- **Связи**: Топ-аспекты и сложные фигуры (например, стеллиумы) на основе максимального веса (weight).
*(В среднем генерируется около 30–45 тематических запросов).*

### 4.2. Поиск по базе знаний и RAG (pgvector / `book_chunks`)
Каждый сгенерированный запрос превращается в вектор с использованием `text-embedding-3-small` и применяется для семантического поиска в базе данных.
- Используется расширение `pgvector`. Учитывается только контекст `category = 'western_astrology'`.
- Устанавливается высокий порог сходства (`score >= 0.72`) для выборки самых точных совпадений из литературы.
- Результаты дедуплицируются, формируя топ-10 кусков текста, наиболее подходящих для расчета.

### 4.3. Генерация инсайтов (Большая языковая модель)
Осуществляется промптинг OpenAI GPT-4o:
- Пробрасываются исходный `Rich JSON` и текстовые фрагменты-обоснования.
- Модель работает в режиме `json_object` с температурой генерации (`temp = 0.4`), возвращая 60–100 объектов инсайтов, разбитых по 12 сферам жизни.
- **Pydantic Validation**: Производится строгая проверка полученных инсайтов. При нарушениях структуры отрабатывает retry с минимальной температурой (0.2). Фатальные ошибки логируются в `uis_errors`.

## 5. Слой синтеза (Layer 3 - Synthesis)
Итоговая сборка профиля пользователя на бэкенде, реализованная строгим кодом (без участия нейросетей):
1. **Группировка**: Карточки инсайтов распределяются по общим системам (напр. `western_astrology`) и вписываются в 12 универсальных сфер (Личность, Ресурсы, Мишление, etc.).
2. **Ранжирование внутри сферы**: Инсайты каждой сферы подвергаются сортировке. Иерархия: Вначале идет `influence_level` (от высокого к низкому), а при равных значениях — `weight` (от 1.0 по убыванию). Рейтинг фиксируется в поле `rank`.
3. **Сохранение в Supabase (`user_insights`)**: Готовый список инсайтов плоским списком (bulk insert) загружается в таблицу `user_insights` для постоянного хранения.
4. **Саммари профиля (`user_portraits`)**: На основании ключевых показателей создается сжатый Identity-портрет: архетип, нарративная роль, тени и дары. Эта выжимка идет в таблицу `user_portraits` и отображается на вкладке "About" приложения.

## 6. Цикл выдачи на клиент
Вся нагрузка по генерации завершена:
- Frontend запрашивает профиль через `GET /api/portraits/:user_id`.
- Из базы моментально извлекается отсортированный по системам и `rank` итоговый объект без дальнейшей дообработки на сервере.
- Контекст `user_insights` и геоданные служат ядром для взаимодействия с Chat Agent (подтягиваясь из `user_memory` для каждого текстового сообщения).
